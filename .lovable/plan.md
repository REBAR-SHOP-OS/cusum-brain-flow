

# Fix Hero Slider — CSS Injection via WordPress Customizer API

## Root Cause (confirmed)

1. The Slider Revolution 7 (SR7) plugin JavaScript fails to populate `<sr7-bg>` elements with background images
2. Our previous "repair script" approach **does not work** because WordPress REST API strips `<script>` tags from page content — the injected script never reaches the browser
3. The slider markup is generated by the SR7 plugin (not page content), so we can't modify slide HTML directly

## Solution

Use the **WordPress Custom CSS API** (`/wp/v2/global-styles` or `/wp-json/wp/v2/settings`) to inject **CSS rules** that set background images on each slide by element ID. WordPress does NOT strip `<style>` content from the customizer.

We know:
- Slide IDs: `SR7_6_1-47`, `SR7_6_1-46`, `SR7_6_1-42`, `SR7_6_1-40`, `SR7_6_1-44`, `SR7_6_1-43`, `SR7_6_1-41`, `SR7_6_1-37`
- Image URLs from `<image_lists>` (8 images in order)

### CSS approach
```css
#SR7_6_1-47-2 { background-image: url(//rebar.shop/wp-content/uploads/2026/01/rebar-estimation-banner-1920-vivid.webp) !important; background-size: cover !important; background-position: center !important; }
#SR7_6_1-46-2 { background-image: url(...) !important; ... }
/* ... one rule per slide bg element */
sr7-slide:first-of-type { opacity: 1 !important; }
```

### Implementation

**Rewrite `supabase/functions/wp-fix-hero/index.ts`:**

1. **GET mode**: Fetch the homepage, parse `image_lists` to extract current slide IDs + image URLs, and check if custom CSS already exists
2. **POST inject**: 
   - Fetch homepage HTML to get current slide IDs and image URLs dynamically (not hardcoded)
   - Build CSS rules mapping each `sr7-bg` element ID to its image URL
   - Use WP REST API endpoint to inject CSS via one of:
     - `PUT /wp/v2/settings` with `custom_css` field (if Woodmart theme supports it)
     - Or prepend a `<style>` block to the page content (WP preserves `<style>` tags, unlike `<script>`)
3. **POST remove**: Remove the injected CSS

**Fallback strategy**: If the settings endpoint doesn't support custom CSS, inject `<style>` tags into the page content instead (WordPress allows inline styles in page content, just not scripts).

### Why this works
- CSS is applied before/during SR7 JS initialization — even if SR7 JS fails completely, the backgrounds will show
- WordPress preserves `<style>` tags in page content (only `<script>` is stripped)
- CSS `!important` overrides any SR7 inline styles
- No dependency on JavaScript execution
